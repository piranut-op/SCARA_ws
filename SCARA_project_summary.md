# SCARA Bot Workspace — Project Summary

This document summarises the entire `~/scara_bot_ws` ROS 2 workspace: the robot, its packages, the simulation and hardware bring-up paths, the camera-driven pick-and-place pipeline, and the calibration that was performed to get the physical arm hitting bottle caps within ~1–2 mm.

---

## 1. Robot

4-DOF SCARA built around two ODrive-driven revolute joints and one MKS-driven prismatic end-effector.

### Mechanical chain
- `Link_1_joint` (revolute, Z-) — shoulder; ODrive motor 1.
- `Link_2_joint` (revolute, Z+) — elbow; ODrive motor 2.
- `ee_joint` (prismatic, Z-) — vertical screw-down end-effector; MKS SERVO motor 3 (14-bit encoder, 16384 counts/rev).
- Workspace: flat surface beneath the arm, viewed by a downward-mounted Intel RealSense camera mounted ≈ 690 mm above `base_link`.

### URDF (`src/scara_robot/urdf/arm.xacro`)
| Joint | Type | Origin in parent (xyz, m) | Axis |
|---|---|---|---|
| Link_1_joint | revolute  | (0, 0, 0.16)                       | (0, 0, −1) |
| Link_2_joint | revolute  | (0.14912, 0.00021, 0.1565)         | (0, 0, +1) |
| ee_joint     | prismatic | (0.14661, −0.035766, −0.105)       | (0, 0, −1) |

The URDF is CAD-exported. The 35.8 mm y-offset on `ee_joint` does **not** exist on the physical robot — see §6 (calibration).

### SRDF planning groups (`src/scara_moveit_config/config/scara_robot.srdf`)
- `arm` — chain `base_link` → `Link_2` via joints `Link_1_joint`, `Link_2_joint`.
- `ee`  — end-effector group with `Link_ee` / `ee_joint`, parented to `Link_2`.
- Named states include `begin_cam_pick` — the camera-clearance home pose used between cycles.

## 3. Capabilities and entry points

There are four largely independent things you can run in this workspace. They share the URDF + SRDF, but each has its own launch file.

### 3.1 Visualisation only
```bash
ros2 launch scara_robot rviz.launch.py
```
URDF + `joint_state_publisher_gui` sliders + RViz. No planner, no controllers.

### 3.2 Gazebo simulation
```bash
ros2 launch scara_robot gazebo.launch.py
```
Spawns the robot in GZ Sim with `ros2_control` and `gazebo_ros2_control`. `GZ_SIM_RESOURCE_PATH` is set from the package share dir — if meshes fail to load, that's the first place to look.

### 3.3 MoveIt 2 demo (fake hardware)
```bash
ros2 launch scara_moveit_config demo.launch.py
```
`move_group` + RViz MotionPlanning panel + fake `JointTrajectoryController`. Used as the planning back-end for the camera pipeline.

### 3.4 Real hardware bring-up (no planner)
```bash
sudo ip link set can0 up type can bitrate 250000
ros2 launch SCARA_pkg scara_hw_bridge.launch.py
```
Brings up `joint_states_bridge` + `newposition` (ODrive bridge for joints 1, 2) + `mks_ee` (MKS bridge for joint 3). Accepts `can_interface:=`, `zero_on_start:=`, `set_bus_foc_on_start:=`.

### 3.5 Camera-driven pick-and-place — the headline application
**Three parallel implementations** exist; the first is the production path, the other two are predecessors / minimal-stack alternatives:

```bash
# (a) Preferred — MoveIt planner + analytical IK + CAN hardware. See §5.
ros2 launch scara_robot cam_pick_and_place.launch.py use_hardware:=true

# (b) Full pure-Python pick-and-place state machine, no MoveIt, talks
#     directly to CAN. See §6.
ros2 launch SCARA_pkg cam_pick_and_place.launch.py

# (c) Camera → IK → motors only (no state machine, no place pose).
#     The arm continuously tracks the detected cap. Useful for IK / camera

ros2 launch SCARA_pkg perception_to_ik.launch.py with_hardware:=true
```

Brings up `ikpos` + `fkpos` + `newposition` — manual IK / FK targeting with no perception, no MoveIt, no end-effector. Predates the camera pipeline; kept for manual joint-target poking.

## 4. Fixed-coordinate pick-and-place (`scara_robot/src/pick_and_place.cpp`)

A self-contained MoveIt demo that runs a pick-and-place cycle between two **hard-coded joint configurations** — no camera, no IK, no perception in the loop. Useful as a known-good smoke test of the planner + controller stack before bringing the camera pipeline online.

### Approach
- Targets are expressed directly in joint space (`Link_1_joint`, `Link_2_joint`, `ee_joint`) via `setJointValueTarget` on the `arm` group, then planned and executed by MoveIt. Bypassing Cartesian IK keeps every waypoint guaranteed reachable.
- The `ee_joint` prismatic axis **is driven through MoveIt here** (unlike the camera pipeline, which routes the screw motion through its own topic — see §9).

### Hard-coded waypoints
| Symbol | Value | Meaning |
|---|---|---|
| `pick_j1, pick_j2`  | `+0.8, +1.0` rad   | Pick pose: front-right quadrant. |
| `place_j1, place_j2`| `−0.8, −1.0` rad   | Place pose: opposite quadrant (1.6 rad / 2.0 rad swing). |
| `ee_up`             | `0.000 m`          | Travel height. |
| `ee_dn`             | `0.060 m`          | Engage stroke (60 mm; 5 mm below URDF limit 0.065 m). |

### Cycle
`approach_pick → engage_pick → retreat_pick → approach_place → engage_place → retreat_place → home_return`

At each `engage_*` step the node:
- Reads `getCurrentPose("Link_ee")` and publishes a coloured cube marker on `/scara_pick_place_markers` (green at pick, orange at place) so RViz shows where the EE actually landed — handy for sanity-checking the URDF + controller stack against a ruler.
- Dwells 1.0 s before retreating; other transitions get a 200 ms settle delay.

Planning parameters: 5 attempts, 5 s budget per plan, velocity/accel scaling 0.4, replanning enabled.

### When to use it
- First bring-up of a freshly built workspace, before plugging in the RealSense.
- Reproducing controller / trajectory-execution problems with no perception variables.
- Validating new ros2_control or MoveIt YAML changes (§ MoveIt → ros2_control wiring in CLAUDE.md).

The chosen joint values are pure waypoints — they have no physical pick location. If you want it to engage at a specific table position, change `pick_j*` / `place_j*` directly (or use the camera variant, which solves IK at runtime).

## 5. Camera-driven pick-and-place — MoveIt variant (`scara_robot`)

Top-level launch: `src/scara_robot/launch/cam_pick_and_place.launch.py`.

```
RealSense ──► detect_bottle_cap (Python, SCARA_pkg)
            │  YOLO detection + 5-second pixel-stability lock
            │  rs2_deproject_pixel_to_point(pixel, depth) → 3-D point
            ▼
        /bottle_cap/detections   (geometry_msgs/PointStamped,
                                  frame_id = camera_color_optical_frame, m)
            │
            ▼
        static_transform_publisher    (base_link → camera_color_optical_frame)
            │
            ▼
        cam_pick_and_place (C++, scara_robot)
            │  tf2 transform → ray-plane intersection at workspace_z_m
            │  analytical 2R IK on (x, y) for q1, q2
            │  MoveIt setJointValueTarget on the "arm" group
            ▼
        JointTrajectoryController → /joint_states
            │
            ├──► joint_states_bridge → /odrive/angle_cmd → ODrive (joints 1, 2)
            └──► /mks/ee_relative_counts → mks_ee (joint 3, screw/unscrew)
```

### Why analytical IK instead of MoveIt's KDL
MoveIt's default KDL inverse-kinematics solver could not reliably solve the planar 2R chain when the tip frame (`Link_ee`) sits on the prismatic axis below `Link_2`. The C++ node therefore computes a closed-form 2R IK on `(x, y)` and sends joint-space targets via `setJointValueTarget`.

### Pick-and-place cycle (per detected cap)
1. **Approach pick:** plan to `(cap_x, cap_y, z_travel_m)`.
2. **Engage pick:** plan to `(cap_x, cap_y, z_pick_m)` (prismatic z-command was disabled during calibration — see §6).
3. **Screw cycle** on motor 3: CCW 7 revolutions (screw down) → 1 s dwell → CW 7 revolutions (unscrew up).
4. Retreat to travel z.
5. Approach place → engage place → screw cycle (same as step 3) → retreat.
6. Return to SRDF named state `begin_cam_pick` (camera-clearance home pose).
7. Cool-down 2 s, then accept the next detection.

Loop is single-threaded with blocking sleeps so joints 1/2 hold position while motor 3 performs the screw motion.

## 6. Camera-driven pick-and-place — `SCARA_pkg` variant

Top-level launch: `src/SCARA_pkg/launch/cam_pick_and_place.launch.py`.

A parallel implementation that **does not use MoveIt**:
- `detect_bottle_cap` (same node) publishes detections.
- `pick_and_place_cam.py` (Python) subscribes, transforms via `cap_to_target.py`, runs analytical IK from `ikpos.py`, and writes joint commands directly through `newposition` / `mks_ee` — same CAN-bus contract as the MoveIt variant.

Use the MoveIt variant by default. The SCARA_pkg variant exists as a no-planner, no-RViz back-up for headless / minimal-stack operation. See `src/README_cam_pick_and_place.md` for the side-by-side comparison.

## 7. CAN bus implementation details

### ODrive (motors 1, 2)
- 11-bit arbitration ID = `(axis_id << 5) | command_id`.
- Each power-on requires:
  1. Full encoder-calibration sequence (cmd `0x007`, data `[0x03, …]`).
  2. Set closed-loop control state (cmd `0x007`, data `[0x08, …]`).
  3. Set position-control mode.
- Performed manually via `setup_scara_gui` before every test session, one motor at a time.

### MKS SERVO (motor 3)
- 11-bit arbitration ID = motor node ID directly (`0x003`).
- Frame: `[cmd, …payload, crc]` where `crc = (node_id + sum(payload)) & 0xFF`.
- Commands used:
  - `0xF5` — absolute move to a 24-bit signed encoder-count target (RPM + accel byte).
  - `0x92` — set current shaft position as count=0 (run at startup when `zero_on_start:=true`).
  - `0x82` — set working mode (5 = Bus FOC; `set_bus_foc_on_start:=true`).
- 14-bit encoder, **16384 counts/rev**. Datasheet "CW = positive count" is from the **shaft-end** view; the operator-side convention is opposite, so the bridge flips signs in software.
- The MKS firmware is factory-calibrated and persists working mode across power cycles. It locks the shaft the instant 24 V is applied — no host enable command needed.

## 8. Calibration procedure and findings

The arm produced a ~5 cm Cartesian error on its first test pick. Decomposed and corrected term-by-term:

### Step 1 — Lateral (y) error: 3.6 cm
**Cause:** `L2_OFF_Y = −0.035766 m` in the CAD-exported URDF does not exist on the physical robot. The IK was compensating for a ghost offset that became a ~3.6 cm spatial error rotating with arm pose.

**Fix:** in the analytical IK (`cam_pick_and_place.cpp`):
```
L2_OFF_X = 0.13661   (was 0.14661)
L2_OFF_Y = 0.0       (was −0.035766)
L2       = L2_OFF_X
BETA     = 0
```

### Step 2 — Radial (x) error: 1 cm, tip falling short
**Cause:** physical L2 was ~10 mm shorter than URDF.
**Fix:** reduce `L2_OFF_X` by 10 mm (0.14661 → 0.13661).

### Step 3 — Residual tangential error after changing home pose
After end-of-cycle home was changed to the camera-clearance pose `(+π/2, +π/2)`, a fresh ~10 mm tangential error appeared. Traced to **gearbox backlash on Link_1**: the arm now approached each target from the opposite direction, loading the backlash slack on the other side.

**Fix:** re-tune `joint1_offset_turns` (motor-turn bias applied inside `newposition.py`). Empirical mapping:
```
1 mm tangential error at 30 cm reach ≈ 0.05 motor turns
```
Final value: `joint1_offset_turns = −0.5`. Both joint offsets are now plumbed through `cam_pick_and_place.launch.py` as launch args.

### Step 4 — Motor 3 jump at first engage
MKS encoder retains absolute zero in flash. Hand back-driving the shaft before power-on leaves the motor at a non-zero count, so the first F5 command yanks it back to flash-zero.

**Fix:** `zero_on_start = true` is now the default. `mks_ee` sends `0x92` at startup to redefine the current shaft position as zero. The screw cycle is symmetric (+delta then −delta), so there is also no drift across many cycles.

### Step 5 — Direction of screw motion
Initial implementation followed the MKS datasheet sign convention (CCW = negative counts), but visually rotated the wrong way because the datasheet's CCW is defined looking at the shaft end face, not from the operator's view.

**Fix:** sign flipped in software. `+delta` = CCW from operator view = screw down.

## 9. End-effector control during calibration

During planar-IK calibration, `ee_joint` was deliberately disabled to isolate joint-1/joint-2 error from any vertical-axis interference:
- `ee_joint` was removed from `setJointValueTarget`'s JointMap in `cam_pick_and_place.cpp`.
- `mks_ee` was temporarily commented out of `scara_hw_bridge.launch.py`.

Once calibration was complete, `mks_ee` was re-enabled, and motor 3 is now commanded only via `/mks/ee_relative_counts` at engage points — never through the joint-trajectory path. This keeps the planar joints fully isolated from the screw motion.

## 11. Achieved performance

After full calibration:
- Cartesian accuracy at the cap: **~1–2 mm**, limited by mechanical play and pixel-level detection granularity.
- Cycle time: detect → pick → screw → place → screw → home in **~15–20 s** per cap (dominated by MoveIt planning + two 3-second screw legs).
- No drift across consecutive picks — each screw cycle is balanced (net zero motor-3 displacement).

## 13. Lessons learned

1. **CAD-exported URDFs cannot be trusted for spatial accuracy.** Mesh-frame offsets (e.g. the 35.8 mm y-offset on Link_2) often reflect SolidWorks reference geometry, not the physical assembly. Verify with `tf2_echo base_link Link_ee` at a known joint configuration and a ruler before tuning anything else.
2. **Backlash interacts with home-pose choice.** Calibration offsets are only valid for the direction of approach used during tuning. Changing the cycle's resting pose silently invalidates a tuned offset until re-tuned.
3. **Datasheet sign conventions need verification.** Both ODrive's `direction_sign` and MKS's CW/CCW definitions reference the motor's own frame, not the operator's view. Confirm direction empirically.
4. **MKS persistent-zero needs explicit handling.** Back-driving the shaft between power cycles will cause the first absolute-position command to snap back to the flash zero unless `0x92 set_zero` is sent at startup.
5. **Decompose Cartesian error into radial vs. tangential** relative to the base→target line — radial points at link length / Link_2 offset; tangential points at Link_1 offset. Fastest way to localise which joint to retune.
6. **Keep the screw axis off the joint-trajectory path.** Driving the prismatic joint through MoveIt's planner alongside the planar joints caused vertical interference during planar calibration. Routing motor 3 through its own `/mks/ee_relative_counts` topic keeps the two control problems independent.
